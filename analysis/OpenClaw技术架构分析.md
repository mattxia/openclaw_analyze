# OpenClaw 技术架构分析

## 一、项目概述
OpenClaw是一个**自托管个人AI助手系统**，支持在用户自有设备上运行，提供跨平台多渠道的AI交互能力。核心设计理念是"本地优先、隐私可控、高度可扩展"，用户可以通过任意常用的即时通讯渠道（WhatsApp/Telegram/Discord/Slack/微信等）与自己的AI助手交互，同时支持设备级能力调用和自定义扩展。

## 三、核心组件职责

### 1. 接入层组件
| 组件类型 | 职责 | 技术实现 |
|---------|------|----------|
| **消息通道** | 对接各类即时通讯平台，实现消息收发，屏蔽平台差异 | 各平台官方SDK/第三方库（grammY for Telegram, discord.js for Discord等） |
| **Web端** | 提供Web控制面板和WebChat交互界面，支持浏览器直接访问 | React + WebSocket |
| **桌面端** | macOS菜单应用，提供语音唤醒、快捷操作、系统级集成 | Swift + 原生API |
| **移动端** | iOS/Android应用，提供移动设备能力（摄像头、位置、通知等） | 原生开发 + WebSocket通信 |
| **CLI工具** | 命令行交互接口，支持系统配置、调试、指令执行 | Node.js + Commander |

### 2. 控制平面层（Gateway核心）
Gateway是整个系统的中枢，运行在用户设备/服务器上，作为统一的控制平面：

| 模块 | 职责 | 核心文件 |
|------|------|----------|
| **消息处理模块** | 消息预处理、去重、权限校验、路由分发、自动回复生成 | `src/auto-reply/` 目录 |
| **代理运行模块** | 运行Pi-Agent核心，管理AI会话上下文，处理工具调用请求 | `src/agents/pi-embedded-runner/` |
| **配置管理模块** | 系统配置、账号配置、通道配置、权限配置的存储和加载 | `src/config/` |
| **会话管理模块** | 多会话隔离、上下文存储、会话生命周期管理 | `src/auto-reply/context.ts` |
| **定时任务模块** | Cron定时任务、自动化工作流、任务调度 | `src/cron/` |
| **Web服务模块** | HTTP/HTTPS服务、WebSocket控制平面、API接口、静态资源服务 | `src/gateway/` |
| **权限控制模块** | 用户配对、权限校验、访问控制、安全策略执行 | `src/acp/` |
| **扩展管理模块** | 插件、技能、扩展的加载、管理、生命周期控制 | `src/hooks/`、`extensions/` 目录 |

### 3. 核心能力层
| 系统 | 职责 | 技术实现 |
|------|------|----------|
| **工具系统** | 提供AI可调用的工具集：浏览器控制、文件操作、系统命令、截图、录屏等 | Chrome CDP协议、系统原生API |
| **技能系统** | 可扩展的能力包，支持第三方服务集成（天气、日历、Github、Notion等） | 标准化SKILL接口，动态加载 |
| **节点系统** | 分布式设备节点管理，支持多设备协同，调用设备本地能力 | WebSocket双向通信、设备权限控制 |
| **模型系统** | 多LLM模型支持、故障转移、token计费、请求限流 | 标准化模型API抽象，支持OpenAI/Anthropic/Qwen等 |

## 四、组件间关系与核心交互流程

### 1. 典型消息处理流程
```
用户消息 → 对应Channel监听器 → 消息格式转换为内部统一格式 → 消息预处理（身份校验/去重/媒体处理）
→ 消息分发（路由到对应会话） → 自动回复模块构建上下文和工具集 → 调用Pi-Agent执行推理
→ 处理工具调用请求（如果有） → 生成回复内容 → 适配到目标Channel格式 → 发送回用户
```

### 2. 工具调用流程
```
AI代理需要调用工具 → Gateway校验工具权限 → 工具路由：
  → 本地工具：直接在Gateway所在设备执行
  → 节点工具：通过WebSocket发送到对应设备节点执行
→ 获取工具执行结果 → 返回给AI代理继续推理
```

### 3. 节点通信流程
```
设备节点启动 → 发现Gateway（本地发现/远程配置） → 配对认证 → 注册设备能力列表
→ Gateway维护节点状态 → 需要调用设备能力时通过WebSocket下发指令 → 节点执行后返回结果
```

## 五、技术栈与架构特点

### 核心技术栈
- **运行时**: Node.js ≥ 22 + TypeScript
- **前端**: React + WebSocket
- **桌面端**: Swift (macOS)
- **移动端**: 原生iOS/Android开发
- **数据存储**: 本地文件系统 + 嵌入式数据库
- **协议**: HTTP/HTTPS + WebSocket + 各通讯平台私有协议

### 架构优势
1. **高度解耦**：通道层、网关层、能力层完全分离，新增通道/能力不影响核心逻辑
2. **可扩展性强**：技能系统、插件系统支持用户自定义扩展
3. **本地优先**：所有数据和计算默认在用户本地设备完成，隐私可控
4. **多设备协同**：支持多设备节点分布式部署，能力互补
5. **安全设计**：默认DM配对机制、权限隔离、沙箱运行，降低安全风险

### 典型部署模式
- **单机模式**: Gateway + 所有能力运行在单台设备（macOS/Linux/Windows）
- **分布式模式**: Gateway运行在服务器，多设备（手机/电脑/智能家居）作为节点接入
- **远程模式**: Gateway运行在云服务器，用户通过加密连接远程访问
